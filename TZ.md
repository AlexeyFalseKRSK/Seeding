# Техническое задание на разработку программного средства  
## «Интеллектуальная система компьютерного зрения для идентификации и анализа посадочного материала на изображениях»

## 1. Общие сведения

**Наименование программного средства:**  
Seeding.

**Вид разработки:**  
разработка настольного программного средства для автоматизированного анализа изображений посадочного материала.

**Область применения:**  
система предназначена для использования при обработке графических материалов, содержащих изображения посадочного материала, с целью его автоматизированного обнаружения, выделения, анализа и документирования результатов.

**Характер разработки:**  
разработка нового программного средства с графическим пользовательским интерфейсом.

---

## 2. Основание для разработки

Основанием для разработки является необходимость создания программного средства, обеспечивающего автоматизацию процессов визуального анализа изображений посадочного материала, снижения трудоёмкости ручной обработки изображений и повышения воспроизводимости результатов анализа.

Разрабатываемая система должна обеспечивать загрузку исходных данных, выполнение анализа изображений с использованием моделей компьютерного зрения, визуальное отображение результатов, а также формирование итоговой отчётной документации.

---

## 3. Назначение разработки

Разрабатываемое программное средство предназначено для решения следующих задач:

- загрузка изображений и PDF-документов, содержащих посадочный материал;
- автоматическое обнаружение объектов посадочного материала на изображении;
- выделение и анализ отдельных частей обнаруженных объектов;
- визуализация результатов обработки в графическом интерфейсе;
- подготовка итогового отчёта по результатам анализа;
- обеспечение возможности поставки программного средства в виде готовой пользовательской сборки.

---

## 4. Цель разработки

Целью разработки является создание настольного программного средства, позволяющего выполнять автоматизированную идентификацию и анализ посадочного материала на изображениях с использованием методов компьютерного зрения.

Результатом разработки должно стать законченное приложение, пригодное для использования конечным пользователем без необходимости изменения исходного кода, с понятным графическим интерфейсом, поддержкой сборки, установки и эксплуатации на целевой рабочей станции.

---

## 5. Требования к программному средству

## 5.1. Общие требования

Программное средство должно быть реализовано в виде настольного приложения с графическим интерфейсом пользователя.

Система должна обеспечивать:

- локальную работу на одном рабочем месте;
- обработку изображений и PDF-документов;
- выполнение анализа изображений на основе обученных моделей компьютерного зрения;
- отображение результатов анализа в интерактивном интерфейсе;
- формирование итогового PDF-отчёта;
- поставку в виде готовой сборки для конечного пользователя.

---

## 5.2. Требования к функциональному составу

В состав программного средства должны входить следующие функциональные подсистемы:

### 5.2.1. Подсистема загрузки данных

Подсистема загрузки данных должна обеспечивать:

- открытие одиночных изображений;
- открытие набора изображений;
- открытие PDF-документов;
- преобразование страниц PDF в изображения для последующей обработки;
- добавление новых файлов в уже открытую рабочую сессию.

### 5.2.2. Подсистема анализа изображений

Подсистема анализа изображений должна обеспечивать:

- запуск модели детекции объектов посадочного материала;
- определение координат ограничивающих рамок найденных объектов;
- определение класса объекта;
- определение вероятности распознавания объекта;
- формирование промежуточных данных для последующего анализа.

### 5.2.3. Подсистема анализа частей объекта

Подсистема анализа частей объекта должна обеспечивать:

- выбор обнаруженного объекта для дальнейшего анализа;
- запуск отдельной модели анализа частей объекта;
- выделение частей объекта внутри ограничивающей рамки;
- определение класса каждой части;
- определение вероятности распознавания каждой части;
- сохранение результатов анализа для последующего отображения и включения в отчёт.

### 5.2.4. Подсистема визуализации

Подсистема визуализации должна обеспечивать:

- отображение исходного изображения;
- отображение найденных объектов на изображении;
- отображение частей объекта;
- отображение списка или дерева найденных элементов;
- отображение служебной информации по текущему изображению и выбранному объекту;
- изменение масштаба отображения;
- переключение между загруженными изображениями или страницами документа.

### 5.2.5. Подсистема преобразования изображений

Подсистема преобразования изображений должна обеспечивать:

- поворот исходного изображения;
- поворот выделенного объекта;
- корректный пересчёт координат ограничивающих рамок после поворота;
- сохранение согласованности результатов анализа после геометрических преобразований.

### 5.2.6. Подсистема формирования отчёта

Подсистема формирования отчёта должна обеспечивать:

- формирование итогового PDF-документа;
- включение в отчёт аннотированных изображений;
- включение в отчёт информации о найденных объектах;
- включение в отчёт информации о частях объектов;
- сохранение отчёта в файловую систему.

---

## 5.3. Требования к пользовательскому интерфейсу

Графический интерфейс программного средства должен обеспечивать удобную работу оператора без необходимости использования командной строки в процессе основной эксплуатации.

Интерфейс должен содержать:

- главное окно приложения;
- область отображения текущего изображения;
- панель или дерево загруженных изображений, страниц или объектов;
- элементы управления запуском анализа;
- элементы управления поворотом и масштабированием;
- элементы управления формированием отчёта;
- элементы отображения статистической и служебной информации.

Интерфейс должен отвечать следующим требованиям:

- быть единообразным по стилю оформления;
- обеспечивать читаемость элементов управления и подписей;
- обеспечивать логичное расположение элементов;
- поддерживать работу в развернутом оконном режиме;
- обеспечивать однозначное понимание состояния программы пользователем.

---

## 5.4. Требования к архитектуре программного средства

Программное средство должно быть реализовано по модульному принципу.

В структуре системы должны быть выделены следующие логические компоненты:

- модуль запуска приложения;
- модуль конфигурации;
- модуль пользовательского интерфейса;
- модуль работы с изображениями;
- модуль работы с PDF-документами;
- модуль инференса моделей компьютерного зрения;
- модуль обработки результатов детекции и классификации;
- модуль формирования отчётов;
- модуль тестирования.

Архитектура должна обеспечивать:

- разделение интерфейсной и вычислительной логики;
- возможность замены или дообучения моделей без переработки общей структуры приложения;
- возможность сопровождения и расширения программного средства;
- минимизацию жёсткой связанности между модулями.

---

## 5.5. Требования к входным данным

Входными данными программного средства являются:

- графические файлы изображений;
- PDF-документы;
- файлы весов моделей компьютерного зрения;
- параметры конфигурации приложения.

Система должна обеспечивать корректную обработку входных данных при условии соответствия их установленным форматам.

---

## 5.6. Требования к выходным данным

Выходными данными программного средства являются:

- визуально отображённые результаты анализа;
- данные о найденных объектах и их частях;
- аннотированные изображения;
- итоговый PDF-отчёт.

---

## 5.7. Требования к надёжности

Программное средство должно обеспечивать устойчивую работу при штатной эксплуатации.

Система должна:

- корректно обрабатывать отсутствие результатов распознавания;
- корректно обрабатывать ошибки загрузки входных файлов;
- корректно обрабатывать отсутствие файлов моделей;
- исключать аварийное завершение при пустых или некорректных данных;
- обеспечивать вывод пользователю понятных диагностических сообщений при возникновении ошибок.

---

## 5.8. Требования к производительности

Программное средство должно обеспечивать выполнение анализа изображения за приемлемое для пользователя время на целевой рабочей станции.

Допускается, что фактическое время обработки зависит от:

- размера изображения;
- количества объектов на изображении;
- используемой модели;
- вычислительных ресурсов рабочей станции.

При этом интерфейс должен сохранять работоспособность и не должен приводить к полной блокировке пользовательского взаимодействия на длительное время без индикации выполняемой операции.

---

## 5.9. Требования к программной платформе

Разработка программного средства должна выполняться с использованием следующих технологий:

- язык программирования Python;
- графическая библиотека для создания пользовательского интерфейса;
- библиотека для обработки изображений;
- библиотека для работы с PDF-документами;
- библиотека или фреймворк для запуска моделей компьютерного зрения;
- средства автоматизированного тестирования;
- средства сборки и упаковки приложения.

Целевая эксплуатационная платформа — персональный компьютер под управлением операционной системы семейства Windows.

---

## 5.10. Требования к тестированию

В составе проекта должны быть предусмотрены средства тестирования, обеспечивающие проверку основных компонентов программного средства.

Тестирование должно включать:

- проверку загрузки входных файлов;
- проверку запуска моделей;
- проверку корректности обработки результатов;
- проверку формирования отчёта;
- проверку устойчивости к ошибочным входным данным.

Для автоматизированной проверки должна использоваться отдельная тестовая подсистема.

---

## 6. Требования к сборке программного средства

## 6.1. Общие требования к сборке

Для программного средства должен быть предусмотрен формализованный процесс сборки.

Сборка должна обеспечивать:

- установку зависимостей проекта;
- формирование исполняемого варианта приложения;
- проверку корректности структуры поставки;
- возможность повторяемого получения рабочего результата на новой машине разработчика.

Процесс сборки должен быть документирован.

---

## 6.2. Требования к разработческой сборке

Разработческая сборка должна обеспечивать:

- получение работоспособного приложения из исходного кода;
- подключение необходимых зависимостей;
- запуск приложения в среде разработчика;
- запуск автоматизированных тестов;
- возможность подключения файлов моделей без внесения изменений в исходный код приложения.

Разработческая сборка должна использовать единый механизм управления зависимостями и запуском проекта.

---

## 6.3. Требования к пользовательской сборке

Для программного средства должна быть подготовлена конечная пользовательская сборка.

Пользовательская сборка должна обеспечивать:

- запуск приложения без необходимости ручного запуска через интерпретатор Python;
- наличие исполняемого файла приложения;
- наличие всех необходимых библиотек и ресурсов;
- работоспособность на целевой операционной системе после стандартного развёртывания;
- отсутствие необходимости редактирования исходного кода для начала эксплуатации.

---

## 6.4. Требования к сборочному процессу

Сборочный процесс должен включать следующие этапы:

1. подготовка исходного кода и структуры проекта;
2. установка зависимостей;
3. подключение ресурсов и моделей;
4. формирование исполняемой сборки;
5. проверка запуска собранного приложения;
6. проверка корректности работы основных функций;
7. формирование конечного комплекта поставки.

Сборочный процесс должен быть воспроизводимым и позволять повторное получение идентичной структуры поставки.

---

## 6.5. Требования к конечному виду поставки

Конечный вид программного средства должен представлять собой законченное настольное приложение.

Поставка должна включать:

- исполняемый файл приложения;
- каталог ресурсов приложения;
- каталог или механизм подключения файлов моделей;
- документацию по запуску и эксплуатации;
- при необходимости — служебные библиотеки и вспомогательные файлы.

Структура поставки должна быть понятной и удобной для конечного пользователя.

Рекомендуемый состав поставки:

- `Seeding.exe` или иной исполняемый файл приложения;
- каталог `models/` для хранения файлов весов моделей;
- каталог `resources/` для служебных ресурсов интерфейса;
- файл инструкции пользователя;
- файл инструкции по первичной настройке.

---

## 6.6. Требования к установке

Программное средство должно поддерживать один из следующих вариантов распространения:

- portable-вариант, не требующий установки;
- установочный пакет с развёртыванием в выбранный каталог.

Независимо от выбранного способа распространения должны соблюдаться следующие требования:

- пользователь должен иметь возможность запустить приложение после минимального числа действий;
- структура поставки не должна требовать ручной перекомпоновки пользователем;
- путь к моделям должен задаваться штатными средствами приложения или конфигурации;
- все основные функции должны быть доступны после корректного размещения файлов поставки.

---

## 6.7. Требования к первичной настройке

После установки или распаковки приложения пользователь должен иметь возможность выполнить первичную настройку программного средства.

Первичная настройка должна включать:

- указание путей к моделям анализа;
- проверку доступности необходимых файлов;
- подготовку рабочей среды приложения;
- сохранение пользовательских параметров запуска.

Процедура первичной настройки должна быть описана в документации.

---

## 7. Требования к документации

В составе программного средства должна быть подготовлена документация, включающая:

- руководство разработчика;
- инструкцию по сборке проекта;
- инструкцию по запуску проекта в среде разработки;
- инструкцию по формированию пользовательской сборки;
- инструкцию пользователя;
- описание структуры каталогов поставки;
- описание порядка подключения моделей.

Документация должна быть достаточной для:

- повторения процесса сборки;
- установки программного средства;
- первичной настройки;
- повседневной эксплуатации;
- базового сопровождения проекта.

---

## 8. Требования к этапам разработки

Разработка программного средства должна выполняться поэтапно.

### Этап 1. Проектирование

На этапе проектирования должны быть выполнены:

- определение состава подсистем;
- разработка структуры проекта;
- выбор технологий реализации;
- проектирование пользовательского интерфейса;
- определение структуры входных и выходных данных.

### Этап 2. Реализация базового функционала

На данном этапе должны быть реализованы:

- загрузка изображений и PDF-документов;
- базовый графический интерфейс;
- запуск детекции объектов;
- отображение результатов детекции.

### Этап 3. Реализация анализа частей объекта

На данном этапе должны быть реализованы:

- выделение частей обнаруженного объекта;
- визуализация результатов анализа частей;
- сохранение результатов анализа.

### Этап 4. Реализация преобразования и редактирования

На данном этапе должны быть реализованы:

- поворот изображений;
- поворот объектов;
- пересчёт координат после преобразований;
- обновление визуального представления результатов.

### Этап 5. Реализация формирования отчёта

На данном этапе должны быть реализованы:

- генерация PDF-отчёта;
- включение изображений и аналитических данных;
- сохранение документа в файловую систему.

### Этап 6. Подготовка сборки и документации

На данном этапе должны быть выполнены:

- настройка сборочного процесса;
- формирование конечного комплекта поставки;
- подготовка инструкции по установке и эксплуатации;
- проведение итоговой проверки работоспособности.

---

## 9. Порядок контроля и приёмки

Программное средство считается соответствующим техническому заданию при выполнении следующих условий:

- приложение запускается на целевой рабочей станции;
- приложение позволяет открывать изображения и PDF-документы;
- приложение выполняет анализ изображений средствами компьютерного зрения;
- приложение отображает результаты анализа в интерфейсе;
